# Rule: shipping automations that other things depend on

Trigger: you are about to deploy or modify a watcher / liveness monitor / matcher over other agents' output, or to write a status field, marker, or structured value that some other automation reads.

- **Self-test a watcher against real data variants before it goes live.** *(mined 2026-07-12 — 5 incidents, 3 with filed bug reports)* A monitor that matches other agents' output must be run, before deployment, against every known format variant of the real data (Korean/English display names, date-prefixed timestamps, renamed keys). Two separate false-stale storms came from a regex that didn't know one legitimate timestamp format. Also: exclude the operator itself from its own watch list, and prove — by running it — that every state that should be silent (completed, blocked, terminated) actually produces zero output. A terminated meeting once got a public progress ping every interval, forever.
- **A value someone else reads is a contract; check the reader, not your memory.** *(mined 2026-07-12 — 4 incidents, incl. a 128-reinjection loop)* When you write or change a status field, completion marker, frontmatter key, or any structured value that a watchdog, parser, gate, or template consumes, open the consumer's code or spec right after writing and confirm the field name, markup, and format match what it actually reads. The worst case on record: a completion record written with the wrong field name was invisible to the scheduler reading it, which re-injected the "unfinished" task 128 times.
